## 1. 项目总览

### 1.1 项目基本信息

| 项目名称 | 校服生产管理小程序 |
|----------|-------------------|
| 项目类型 | 微信小程序（微信云开发） |
| 开发周期 | 约 10~12 周（约 3 个月） |
| 团队规模 | 产品经理 1 人、开发 1 人（前端页面+云函数）、UI 设计 1 人、测试 1 人 |
| 交付形式 | 微信小程序可体验版本（云开发版，无需自建服务器） |

### 1.2 技术选型建议

| 技术层级 | 选型方案 | 说明 |
|----------|----------|------|
| 前端小程序 | 原生微信小程序（MINA 框架） | 使用 WXML + WXSS + JS，稳定性高，社区成熟 |
| 后端服务 | 微信云开发（云函数） | 无需自建服务器，后端逻辑运行在云端 |
| 数据库 | 微信云数据库（MongoDB 风格） | 云端托管，支持实时推送，无需运维 |
| 文件存储 | 微信云存储 | 云端存储物料照片和出库单照片 |
| 消息推送 | 微信订阅消息 | 各角色待办通知使用微信订阅消息触达 |
| 登录鉴权 | 账号+密码自主鉴权 | 员工账号由老板创建，账号+密码登录，云函数中校验密码（密码加密存储） |

> **说明**：微信云开发提供了完整的 Serverless 后端能力（云函数 + 云数据库 + 云存储），无需额外部署和维护服务器，可大幅降低开发和运维成本，非常适合此类中小型业务系统。

> **关于配额**：云开发免费基础版支持每日约 1000 次云函数调用和 5GB 存储，如使用量超出可升级按量付费套餐。

## 3. 数据库设计概要

### 3.1 ER 关系图（核心实体）

```
员工 ────1:N───→ 入库单 ────────→┐
  │                              ├→ 原材料库存 ←┌──→ 出库单
  │                              │                 │
  │                              │     ┌────────────┴────────────┐
  │                              │     ▼                          ▼
  │                              │  裁剪来料确认单        车间来料确认单
  │                              │     │                          │
  │                              │     ▼                          │
  │                              │  裁剪单                      │
  │                              │     │                          │
  │                              │     ▼                          │
  │                              │  车间确认单 ◀─────────────────┘
  │                              │     │
  └───────────────→ 供应商    加工单 ──→ 成品确认单
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
               成品库存      成品出库单         通知记录
```

### 3.2 云数据库集合设计（概要）

微信云数据库为 NoSQL 文档型数据库，以下为各集合的字段设计。

> **集合命名规范**：全部使用小写，单词间用下划线分隔，如 `employee`、`raw_inbound_order`。

```javascript
// employee（员工集合）
{
  _id: ObjectId,
  name: String,          // 姓名
  account: String,        // 登录账号
  password: String,      // 密码（加密存储，MD5/SHA256）
  role: String,          // 原材料管理员 / 裁剪管理员 / 车间管理员 / 成品管理员 / 老板
  status: Number,        // 1=正常 0=禁用
  created_at: Date
}

// supplier（供应商集合）
{
  _id: ObjectId,
  name: String,          // 供应商名称
  contact_name: String,   // 联系人姓名
  contact_phone: String,  // 联系电话
  updated_at: Date,
  created_at: Date
}

// raw_material_stock（原材料库存集合）
{
  _id: ObjectId,
  category_one: String,   // 一级分类：布料 / 辅料
  category_two: String,   // 二级分类
  total_quantity: Number, // 总库存数量
  unit: String,           // 计量单位（布料默认"米"，辅料默认"个"，可按实际情况调整）
  warning_threshold: Number, // 库存预警阈值（低于此值触发预警通知）
  updated_at: Date
}
// 唯一索引：category_one + category_two

// raw_inbound_order（原材料入库单集合）
{
  _id: ObjectId,
  order_no: String,       // 订单编号 RK-YYYYMMDD-HHMMSS
  supplier_id: ObjectId,   // 关联供应商
  creator_id: ObjectId,    // 创建人
  material_details: Array, // 物料明细 [{category_one, category_two, quantity, unit}]（unit与raw_material_stock.unit对应）
  photos: Array,           // 照片URL数组
  remark: String,          // 备注
  status: String,          // 已完成 / 已取消
  created_at: Date
}

// raw_outbound_order（原材料出库单集合）
{
  _id: ObjectId,
  order_no: String,            // 订单编号 CK-YYYYMMDD-HHMMSS
  creator_id: ObjectId,
  target_type: String,         // cutting（路径A）/ workshop（路径B）
  target_admin_id: ObjectId,   // 目标裁剪/车间管理员ID
  material_details: Array,    // 物料明细 [{category_one, category_two, quantity, unit}]
  photos: Array,
  remark: String,
  status: String,             // 待确认 / 已完成 / 已取消
  created_at: Date
}

// cutting_incoming_confirm（裁剪来料确认单集合）
{
  _id: ObjectId,
  outbound_order_id: ObjectId, // 关联出库单
  cutting_admin_id: ObjectId,
  confirm_time: Date,
  status: String,             // 待确认 / 已确认 / 有问题
  problem_desc: String,        // 问题描述
  updated_at: Date
}

// workshop_incoming_confirm（车间来料确认单集合）
{
  _id: ObjectId,
  outbound_order_id: ObjectId,
  workshop_admin_id: ObjectId,
  confirm_time: Date,
  status: String,            // 待确认 / 已确认 / 有问题
  problem_desc: String,
  updated_at: Date
}

// cutting_order（裁剪单集合）
{
  _id: ObjectId,
  order_no: String,           // 订单编号 CJ-YYYYMMDD-HHMMSS
  incoming_confirm_id: ObjectId, // 关联裁剪来料确认单
  outbound_order_id: ObjectId,   // 直接关联原材料出库单（便于追溯原始物料来源）
  cutting_admin_id: ObjectId,
  material_actual_usage: Array,  // 实际物料消耗量 [{category_two, quantity, unit}]（unit与raw_material_stock.unit对应）
  plan_clothes_detail: Array, // [{size, gender, style, school, quantity}]
  target_workshop: String,
  status: String,             // 待确认 / 已确认 / 已退回
  created_at: Date,
  updated_at: Date
}

// workshop_order_confirm（车间确认单集合）
{
  _id: ObjectId,
  cutting_order_id: ObjectId,
  outbound_order_id: ObjectId,  // 直接关联原材料出库单（便于一步追溯原始物料来源，与cutting_order.outbound_order_id一致）
  workshop_admin_id: ObjectId,
  confirm_time: Date,
  status: String,            // 待确认 / 已确认 / 已退回
  return_reason: String,
  created_at: Date,
  updated_at: Date
}

// processing_order（加工单集合）
{
  _id: ObjectId,
  order_no: String,           // 订单编号 JG-YYYYMMDD-HHMMSS
  source_type: String,        // cutting（路径A）/ workshop（路径B）
  workshop_confirm_id: ObjectId,              // 路径A：关联车间确认单
  workshop_incoming_confirm_id: ObjectId,     // 路径B：关联车间来料确认单
  workshop_admin_id: ObjectId,
  // 路径A：计划件数来自 cutting_order.plan_clothes_detail，自动填充
  // 路径B：计划件数由车间管理员在加工登记时手工填写
  plan_quantity: Array,      // [{size, quantity}]
  actual_quantity: Array,    // [{size, quantity}]
  loss_rate: Array,          // [{size, rate}]
  accessory_usage: Array,     // 辅料消耗量 [{category_two, quantity}]（数量单位按辅料计量）
  // 路径A：SKU来自 cutting_order，自动填充
  // 路径B：SKU由车间管理员在加工登记时手工填写
  gender: String,            // 男 / 女 / 男女同款
  style: String,             // 款式
  school: String,            // 学校
  status: String,            // 待确认 / 已完成
  confirm_time: Date,        // 确认时间（状态变为"已完成"时填充）
  created_at: Date,
  updated_at: Date
}

// finished_product_confirm（成品确认单集合）
{
  _id: ObjectId,
  processing_order_id: ObjectId,
  finished_admin_id: ObjectId,
  confirm_time: Date,
  actual_quantity: Array,    // 实际确认入库数量 [{size, quantity}]，与加工单的actual_quantity可能不同，用于记录差异
  status: String,             // 待确认 / 已确认 / 有问题
  problem_desc: String,
  created_at: Date,
  updated_at: Date
}

// finished_product_stock（成品库存集合）
{
  _id: ObjectId,
  gender: String,             // 男 / 女 / 男女同款
  style: String,
  school: String,
  size: String,
  quantity: Number,
  updated_at: Date
}
// 唯一索引：gender + style + school + size

// finished_outbound_order（成品出库单集合）
{
  _id: ObjectId,
  order_no: String,           // 订单编号 CC-YYYYMMDD-HHMMSS
  processing_order_id: ObjectId,  // 关联加工单
  // 出库明细：按SKU维度记录，与成品库存维度（性别+款式+学校+尺码）保持一致
  outbound_details: Array[{    // [{gender, style, school, size, quantity}]
    gender: String,           // 男 / 女 / 男女同款
    style: String,            // 款式
    school: String,           // 学校
    size: String,             // 尺码
    quantity: Number          // 本SKU出库数量
  }],
  destination: String,
  photos: Array,          // 出库现场照片
  creator_id: ObjectId,
  status: String,            // 已完成 / 已取消
  created_at: Date
}

// notification（通知记录集合）
{
  _id: ObjectId,
  receiver_id: ObjectId,     // 接收人（个人通知时填写，角色级通知可为空）
  role: String,             // 接收角色（用于库存预警等角色级通知，如"老板"；触发时查询该角色下所有员工，逐个创建通知记录）
  type: String,             // 通知类型：cutting_problem / workshop_problem / cutting_return / processing_submit / product_problem / inventory_warning
  title: String,
  content: String,
  related_order_id: ObjectId,
  is_read: Number,          // 0=未读 1=已读
  created_at: Date
}

// system_option（系统选项集合）
{
  _id: ObjectId,
  type: String,             // school / category_two / size / style / workshop / destination（供应商由supplier集合独立管理）
  name: String,
  creator_id: ObjectId,
  created_at: Date
}
```

> **数据库权限规则**：所有数据集合建议设置"仅创建者及管理员可读写"，通过云函数统一处理权限校验，不在小程序端直接暴露读写权限。

## 6. 关键技术难点与应对策略

| 难点 | 说明 | 应对方案 |
|------|------|----------|
| **库存一致性** | 多人同时操作可能导致库存数量不准确 | 云数据库事务保证；使用原子增减操作（`inc`）；出库前查询+校验双重确认 |
| **订单状态机** | 各单据状态流转复杂，需严格控制状态变更 | 云函数内统一状态机服务，每笔状态变更记录操作日志，禁止跨状态跳跃 |
| **路径B数据兼容性** | 路径B跳过裁剪环节，SKU信息来源与路径A不同 | `processing_order` 增加 `source_type` 区分来源；路径B在加工登记时强制填写 gender/style/school；`workshop_incoming_confirm_id` 字段支持路径B来源追溯 |
| **微信订阅消息** | 用户需主动订阅才能收到通知，否则无法推送 | 引导用户在关键操作完成后点击订阅；提供消息中心页面作为兜底查看入口 |
| **图片上传与压缩** | 小程序照片可能过大，影响加载速度和存储成本 | 前端使用 Canvas 压缩（压缩至 80KB 以内）；上传至微信云存储 |
| **数据权限隔离** | 各角色只能查看自己有权访问的数据 | 云函数统一权限校验，查询时自动拼接 `creator_id` / `target_admin_id` 等过滤条件 |
| **损耗率计算** | 不同尺码损耗率需按比例计算，精度要求高 | 云函数统一计算逻辑，计算公式固化在代码中，不暴露给用户修改 |
| **多维度库存查询** | 成品库存按性别+款式+学校+尺码四维筛选，查询性能要求高 | 对这四个字段建组合索引；成品库存按SKU管理，数据量有限，直接查询性能无压力 |
| **历史数据迁移** | 客户可能有现有 Excel 数据需导入 | 提供标准化 Excel 模板；开发导入脚本，数据清洗后通过云函数批量写入云数据库 |

## 7. 测试策略

### 7.1 测试分层

| 测试类型 | 覆盖范围 | 介入时间 | 负责角色 |
|----------|----------|----------|----------|
| 单元测试 | 库存计算、损耗率计算、状态机流转 | 开发阶段 | 开发 |
| 接口测试 | 所有云函数接口（含边界值、权限校验） | 开发完成提测时 | 测试 |
| 集成测试 | 路径A/路径B全流程端到端 | 每阶段联调期 | 测试 |
| UI 验收测试 | 页面交互、字体大小、触控区域 | 阶段四优化期 | 产品+客户 |
| 性能测试 | 列表加载时间、并发登录 | 阶段四上线前 | 测试 |

### 7.2 回归测试要点

每次 Bug 修复后必须回归以下关键路径：

1. **路径A完整流程**：登录→入库→出库（裁剪）→来料确认→裁剪加工→车间确认单确认→加工登记→成品确认→成品出库
2. **路径B完整流程**：登录→入库→出库（车间）→车间来料确认→加工登记→成品确认→成品出库
3. **权限隔离**：各角色登录后只能看到自己模块，切换角色后数据不泄露
4. **库存校验**：出库数量不得超出当前库存

## 附录：全面逻辑审查说明

本节记录本次全面审查中发现的逻辑硬伤及修复理由，供后续开发参考。

### A.1 路径B数据链完整性

**发现**：PRD 中路径B跳过了裁剪环节，但以下问题会影响系统运行：

1. `processing_order` 缺少区分路径来源的字段，路径A/B 共用同一集合但无法区分来源
2. 路径B 缺少 `workshop_incoming_confirm_id` 字段，无法关联到原始来料确认单
3. 路径B 缺少性别/款式/学校字段，加工登记时无 SKU 信息来源，成品入库时无法按维度入库
4. 路径B 缺少来源追溯链路，成品出库时无法知道该加工单来自哪条出库单

**修复**：
- `processing_order` 增加 `source_type`（cutting/workshop）区分来源
- 增加 `workshop_incoming_confirm_id` 支持路径B来源追溯
- 路径B 在加工登记时强制填写 gender/style/school 字段
- `raw_outbound_order` 增加 `target_type`（cutting/workshop）标识出库目标类型

### A.2 裁剪单物料溯源

**发现**：路径A中，裁剪管理员看到的是来料确认单（关联出库单），但裁剪单本身只关联了来料确认单。如果需要查看原始出库单的全部物料信息，需要两级跳转（裁剪单→来料确认单→出库单）。

**修复**：在 `cutting_order` 中增加 `outbound_order_id` 字段，直接关联原材料出库单，实现一步追溯。

### A.3 通知触发场景完整性

**发现**：PRD 第6章通知推送规则列出了6类触发场景，但实施计划第8周只写了"5类"。经逐一核对，应为6类。

**修复**：修正为6类，并在通知记录集合中增加 `role` 字段，用于库存预警等角色级通知（不指定具体人，而是发给某一角色的所有成员）。

### A.4 员工编辑功能

**发现**：PRD 4.5.2 明确有编辑员工功能，但页面清单缺少编辑页，云函数清单虽有 `employee/update`，但排期未明确。

**修复**：新增 `/pages/boss/employee/edit` 员工编辑页面，并在第2周排期中明确"员工管理云函数（CRUD）"包含编辑功能。

### A.5 路径B无专属加工登记入口

**发现**：PRD 中路径B的"加工登记"操作描述为"选择裁剪单"，但路径B没有裁剪单。如果路径A和路径B共用同一入口，路径B用户将无法选择任何裁剪单，导致功能不可用。

**修复**：新增 `/pages/workshop/processing/add-b` 路径B专属加工登记页面，与路径A共用同一云函数 `workshop/processingAdd`，前端根据用户角色和场景分流到不同页面。

### A.6 登录鉴权描述与技术实现的矛盾

**发现**：技术选型表中写"登录鉴权：微信云开发自带鉴权，无需手动JWT"，但实际设计中：
- `employee` 集合包含 `password` 字段
- 云函数清单包含 `auth/login`（账号+密码校验）
- PRD 明确要求"账号+密码登录"

三者自相矛盾。云开发自带鉴权是微信一键登录，不需要密码，也不会有 `employee` 集合中的密码字段。

**修复**：将技术选型表中的"微信云开发自带鉴权"改为"账号+密码自主鉴权"，明确员工账号由老板创建，密码加密存储，云函数中校验密码。

### A.7 供应商管理数据流断裂

**发现**：`supplier` 集合有完整字段设计，但：
- 云函数清单中完全没有供应商的增删改查
- 老板系统设置页面没有供应商管理入口
- 入库云函数虽有"含新增供应商逻辑"，但没有 `supplier/add` 云函数支撑

供应商集合会被创建但永远无法正常写入数据。

**修复**：
- 云函数清单新增 `supplier/list`、`supplier/add`、`supplier/delete`
- 老板系统设置页面新增供应商管理 Tab `/pages/boss/settings/supplier`

### A.8 成品出库明细维度缺失

**发现**：`finished_outbound_order` 的出库字段设计为单一 `outbound_quantity`，但：
- 成品库存是多 SKU 的（按性别+款式+学校+尺码四维）
- 成品出库 PRD 要求"选择加工订单→系统自动填充尺码/性别/款式/学校→填写出库数量"
- 单一数量字段无法对应多 SKU 出库场景

**修复**：`finished_outbound_order` 的出库明细改为 `outbound_details: Array[{gender, style, school, size, quantity}]`，与 `finished_product_confirm` 的 `actual_quantity` 维度保持一致。

### A.9 页面清单序号重复

**发现**：页面清单中供应商管理页（第26项）和路径B加工登记页都编号为`26`，导致后续所有序号错位。

**修复**：重新编号，供应商管理为26，路径B加工登记为27，消息通知为28，通用弹窗为29。

### A.10 库存预警无触发机制说明

**发现**：关键技术难点中列出"库存预警"但未说明触发方式，是定时任务还是实时触发容易引起歧义。

**修复**：补充说明为"定时任务触发（建议每小时执行一次，扫描所有原材料库存，低于警戒值的物料自动创建通知记录）"。

### A.11 system_option.type 冗余

**发现**：`system_option` 集合的 `type` 字段枚举包含 `supplier`，但供应商管理已独立为 `supplier` 集合、云函数和独立页面，`system_option` 中的 `supplier` 类型属冗余设计。

**修复**：从 `system_option.type` 枚举中移除 `supplier`，并在注释中注明"供应商由 supplier 集合独立管理"。

### A.12 option云函数缺少编辑功能

**发现**：`option` 模块云函数清单只有 `list/add/delete`，缺少 `option/update`。但系统设置中选项名称需要能修改（写错时不能只能删重建），老板系统设置页的编辑功能无法实现。

**修复**：云函数清单新增 `option/update`（编辑选项名称）。

### A.13 原材料出库单取消功能缺失

**发现**：`raw_outbound_order.status` 枚举包含"已取消"，PRD 出入库明细章节也提到"已取消"状态，但云函数清单中没有取消出库单的接口，且取消时需将已扣减的库存回滚。

**修复**：云函数清单新增 `raw/outboundCancel`（取消出库单，含库存回滚和权限校验）。

### A.14 原材料库存缺少计量单位

**发现**：`raw_material_stock.total_quantity` 为纯数字，布料按"米"计量，辅料按"个/卷"计量，不同类型物料混在一起无法区分，易产生歧义。

**修复**：`raw_material_stock` 新增 `unit` 字段（布料默认"米"，辅料默认"个"），入库/出库时与物料分类联动。

### A.15 原材料库存缺少预警阈值字段

**发现**：库存预警通知依赖预警阈值判断，但 `raw_material_stock` 中无 `warning_threshold` 字段，预警云函数无数据来源。

**修复**：`raw_material_stock` 新增 `warning_threshold: Number` 字段（低于此值触发预警通知）。

### A.16 notification.type 缺少枚举定义

**发现**：`notification.type` 为字符串字段，但文档未定义具体枚举值，开发时取值不统一，消息列表无法按类型筛选。

**修复**：补充 type 枚举值：`cutting_problem`、`workshop_problem`、`cutting_return`、`processing_submit`、`product_problem`、`inventory_warning`。

### A.17 material_details 缺少计量单位

**发现**：`raw_inbound_order` 和 `raw_outbound_order` 的 `material_details` 数组无 `unit` 字段，但 `raw_material_stock` 有 `unit`。如果同一物料以不同单位入库，库存累加会产生数量级错误。

**修复**：在 `material_details` 数组元素中增加 `unit` 字段，与库存 `unit` 对应。

### A.18 supplier 云函数缺少 update

**发现**：第11周任务写"供应商/学校/分类/尺码等 CRUD"，明确包含供应商编辑，但云函数清单 `supplier` 模块只有 `list/add/delete`，缺少 `update`。

**修复**：云函数清单新增 `supplier/update`（编辑供应商信息）。

### A.19 workshop_order_confirm 缺少直接物料追溯字段

**发现**：`workshop_order_confirm` 只有 `cutting_order_id`，追溯到原始出库单需要两级跳转（车间确认单→裁剪单→出库单）。为与裁剪单设计保持一致，应增加 `outbound_order_id` 直接关联。

**修复**：`workshop_order_confirm` 新增 `outbound_order_id: ObjectId` 字段。

### A.20 finished_product_confirm 缺少实际入库数量记录

**发现**：成品管理员确认时需要填写实收数量（与加工单的 actual_quantity 可能不同），但 `finished_product_confirm` 无此字段，差异无法记录。

**修复**：`finished_product_confirm` 新增 `actual_quantity: Array[{size, quantity}]` 字段。

### A.21 库存初始化机制缺失

**发现**：第3周无库存初始化说明。若系统上线前已有库存，无初始化机制。

**修复**：第3周任务补充"库存初始化（如有初始库存，由老板提前录入）"。

### A.22 material_actual_usage 和 accessory_usage 缺少字段结构

**发现**：`cutting_order.material_actual_usage` 无结构说明，`processing_order.accessory_usage` 注释写的是 `[{category_two, usage}]`，字段名是 `usage` 与其他集合的 `quantity` 不一致，且无单位说明。

**修复**：`material_actual_usage` 补充结构为 `[{category_two, quantity, unit}]`；`accessory_usage` 字段名保持 `usage`，结构修正为 `[{category_two, quantity}]`（注明数量单位按辅料计量）。

### A.23 三个集合缺少 updated_at 字段

**发现**：`cutting_incoming_confirm`、`workshop_incoming_confirm`、`supplier` 三个集合缺少 `updated_at` 字段，其他操作类集合均有此字段。

**修复**：三个集合各增加 `updated_at: Date` 字段。

### A.24 finished_outbound_order 缺少 photos 字段

**发现**：`raw_inbound_order` 和 `raw_outbound_order` 均有 `photos` 字段记录现场照片，成品出库同样需要留证，但 `finished_outbound_order` 缺少此字段。

**修复**：`finished_outbound_order` 新增 `photos: Array` 字段。

### A.25 四个订单类集合缺少 created_at/updated_at 字段

**发现**：`cutting_order`、`workshop_order_confirm`、`processing_order`、`finished_product_confirm` 四个集合缺少 `created_at` 和 `updated_at`，其他集合均有。

**修复**：四个集合各增加 `created_at: Date` 和 `updated_at: Date` 字段。

### A.26 库存初始化缺少独立云函数

**发现**：第3周有"库存初始化"任务描述，但无对应云函数。`inboundAdd` 执行累加，若系统中无该物料记录会创建多条重复库存，无法唯一定位。

**修复**：云函数清单新增 `raw/stockInit`（库存初始化）；第3周排期补充对应任务。

### A.27 raw_material_stock 缺少唯一性约束说明

**发现**：`finished_product_stock` 有唯一索引注释，但 `raw_material_stock` 没有。`raw_material_stock` 维度为 `category_one + category_two`，同一物料不应有多条记录，需明确说明。

**修复**：`raw_material_stock` 注释补充 `// 唯一索引：category_one + category_two`。

### A.28 processing_order 缺少 confirm_time 字段

**发现**：`workshop_order_confirm` 和 `finished_product_confirm` 均有 `confirm_time`，但 `processing_order`（加工单）从"待确认"变为"已完成"时缺少确认时间节点，影响流转时间轴展示。

**修复**：`processing_order` 新增 `confirm_time: Date` 字段（状态变为"已完成"时填充）。

### A.29 出入库明细页面角色描述不精确

**发现**：页面清单"出入库明细"适用角色写"全部"，但 PRD 权限矩阵明确成品管理员和老板仅可查看，不是全部角色都能操作。

**修复**：修正为"原材料/裁剪/车间（操作）/成品/老板（查看）"。

### A.30 库存初始化缺少页面入口

**发现**：第3周有 `raw/stockInit` 云函数和"库存初始化（老板录入）"任务，但页面清单无对应页面，云函数无法被调用。

**修复**：页面清单新增 `/pages/boss/settings/stock-init` 库存初始化页面（老板专用）。
